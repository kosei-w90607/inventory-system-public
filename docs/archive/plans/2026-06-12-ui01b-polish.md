# UI-01b 商品登録・修正 画面デザイン polish

## Risk

Risk: R3

Reason:
operator-facing form / 一覧 screen の見た目・操作性を変える。ページヘッダー規約、廃番 badge 方針、フォームのセクション分割と read-only / 必須表示、廃番切替の確認ダイアログ、保存成功 toast を追加する。runtime contract（generated command / DTO / route / search params）と DB schema は変更しない。利用者操作と視認性に直接影響し、Windows native L3 で確認する範囲のため R3。

## Goal

UI-01b 商品登録・修正と隣接する UI-01a 一覧 / UI-06a 在庫照会の見た目を、非IT高齢オペレータが入力順に迷わず読める Windows 業務アプリの視覚言語へ揃える。ページヘッダー規約を 3 画面で統一し、一覧の廃番表示を 6 列 + 商品名セル内 badge に整理し、フォームを 4 セクションに分割して read-only と必須項目を明示し、廃番切替に確認ダイアログ、保存成功に toast を追加する。

## Scope

- ページヘッダー規約: UI-01a 一覧 / UI-01b フォーム（通常 + edit-error 分岐）/ UI-06a 在庫照会の h1 を `text-2xl font-semibold` + header wrap に統一する。
- 一覧 ProductTable: 状態列を廃止して 6 列にし、廃番のみ商品名セル内に `Badge variant="secondary"` を出し、廃番行を `text-muted-foreground` にする。「表示中」badge は廃止する。
- フォーム ProductForm: 4 セクション（識別 / 分類と取引先 / 価格 / 在庫）に分割し、各セクション見出しを h2 `text-xl font-semibold` + Separator にする。read-only 入力を `readOnly` + `bg-muted` で示す。
- 必須項目（商品名・部門・売価・原価、create 時は初期在庫）のラベルに `（必須）` を付ける。
- 廃番切替に確認ダイアログ `DiscontinueConfirmDialog`（「廃番にする」のみ確認、「表示に戻す」は直接実行）を追加する。
- 保存成功 toast を create / edit 両 mutation の onSuccess で発火する。
- 上記の設計判断を SCREEN_DESIGN.md / 51-ui-product-form.md / 50-ui-product-list.md の source docs に昇格する。

## Non-scope

- generated command / DTO / route / search params の変更。
- DB schema / migration の変更。
- 数量単位 select の readOnly 化（`disabled` 維持）。
- 一覧の部門列削除（L3 で密度過多なら次候補とする）。
- 新規取引先の inline 作成、商品コード手入力、edit での stock_unit / stock_quantity 変更（既存 UI-01b Non-scope を継続）。
- 新規 npm 依存の追加（AlertDialog / Separator / sonner は導入済み）。

## Review Response

### Codex PR #95 Round 対応

**P2（Codex 指摘、accept 確定）**: `StockUnitField` の自動提案が `onPosStockSyncChange` 経由で `onPosSyncTouchedChange(true)` を立てていた。その結果 pcs→cm→pcs と操作しても touched=true が残り `pos_stock_sync:false` が payload に乗る不整合が成立していた（`stock_unit:"pcs"` + `pos_stock_sync:false`）。

**修正内容**:
- `StockUnitField.tsx`: `onPosStockSyncSuggest: (checked: boolean) => void` prop を追加（touched を立てない自動提案専用経路）。単位変更時は `mode === "create" && !posSyncTouched` のとき `onPosStockSyncSuggest` を呼ぶ（pcs→cm で false、cm→pcs で true）。checkbox の onChange は従来どおり `onPosStockSyncChange` を使う。
- `ProductForm.tsx`: `onPosStockSyncSuggest={(checked) => { update("posStockSync", checked); }}` を配線（`onPosSyncTouchedChange` は呼ばない）。
- テスト 5 本追加（`StockUnitField.test.tsx` +4、`ProductForm.test.tsx` +1）。Red → Green で動作確認済み。
- 設計書更新: `51-ui-product-form.md §7.5 Create step 4` + `§7.9 変更履歴`。
- 全 gate 通過: `npm test`（283 tests）/ `typecheck` / `lint` / `format:check` / `doc-consistency-check.sh` / `generate_traceability --check`。

### PR #95 Codex P2 修正の検証中に発見した lost update バグ修正

**発見経緯**: commit `3aa0341`（Codex P2 対応: POS 同期自動提案と利用者 override の分離）の検証中に orchestrator が実証。

**バグ内容**: `ProductForm.tsx` の `update()` が props の `values` snapshot から `{ ...values, [key]: value }` を作り `onValuesChange` に渡していた。`StockUnitField.handleUnitChange` は同一 tick 内で `onStockUnitChange` と `onPosStockSyncSuggest` を連続で呼ぶため、両方の `update()` 呼び出しが同じ古い `values` snapshot をベースに state を作り、後勝ちの `posStockSync` 更新が `stockUnit` 更新を上書きしていた。`3aa0341` で suggest 経路が touched を立てなくなったことにより毎回 suggest が発火し、create form で cm を選んでも pcs に戻り続ける問題として顕在化。

**修正内容**:
- `ProductForm.tsx`: `onValuesChange` prop 型を `React.Dispatch<React.SetStateAction<ProductFormValues>>` に変更。`update()` を functional update `onValuesChange((prev) => ({ ...prev, [key]: value }))` に変更。`ProductFormPage.tsx:227` の `onValuesChange={setValues}` は `useState` setter をそのまま渡しているため変更不要。
- `ProductForm.test.tsx`: stateful wrapper（`useState` + `onValuesChange={setValues}`）を使う統合テスト 3 本を追加（lost update 検出）。既存テストの updater function capture を `applyUpdater` ヘルパーで実値に変換する形に更新（検証意図を維持）。
- 設計書更新: `51-ui-product-form.md §7.9 変更履歴` に prop 型変更を記録。
- 全 gate 通過: `npm test`（286 tests / 48 files）/ `typecheck` / `lint` / `format:check` / `doc-consistency-check.sh` / `generate_traceability --check`（ERROR 0 / WARN 1 = REQ-403 既知許容）。
