## 4. memory 上書き draft (commit 6 で Write)

**対象**: `~/.claude/projects/-home-kosei-Projects-inventory-system/memory/feedback-plans-sync-commit-milestone-only.md`

**新本文**:

```markdown
---
name: Plans.md は進捗発生即 update + 専用 commit は出さない（同 PR 内 verify commit に同梱）
description: hook リマインダで Plans.md 編集はリアルタイム、commit は同 PR の最終 verify commit に同梱、次 PR 持ち越し廃止
metadata:
  type: feedback
---
Plans.md の更新（SSOT 維持）は hook リマインダ受領時にリアルタイムで行うが、**専用 commit は出さない**。同 PR 内の最終 verify/sync commit (`chore: verify + plans sync ...`) に同梱して 1 commit に集約する。

**Why (2026-05-13 更新):** 旧運用 (2026-04-22 〜 2026-05-12) の「次の機能 PR の第 1 commit で post-PR sync を吸収」は、PR 文脈と無関係な archive 作業が新 PR の commit history を汚染し、bisect 効率と PR review 集中度を下げていた。さらに「Plans 更新タイミングを決め込む」運用は柔軟性に欠け、Plans を見たときに最新でないという SSOT 機能不全を起こしていた。新運用は「進捗発生即 update + 同 PR の verify commit 末尾に同梱」で、PR 単位の自己完結性 + リアルタイム性を両立する。

**How to apply:**
- `PostToolUse:Edit` / `PostToolUse:Bash` hook の Plans.md 更新リマインダ受領 → 即 Edit、ただし単独 commit しない
- working tree に溜めて、PR の **verify/sync commit**（品質ゲート pass 後の最終 commit）で `Plans.md` を含めて 1 commit に集約
- verify commit のメッセージ形式: `chore(<scope>): verify + plans sync (<PR-NN> open|merged)`
- post-merge の archive 作業（`docs/plans/` → `docs/archive/plans/` 移動 + 相対パス変換）は、マージ後の即セッションで Plans.md L17-19 を `[x]` 化 + archive 移動を **1 commit (`docs(plans): post-PR #NN archive`) として独立 push**。これは PR ではなく main 直 push を許容する例外（単独開発プロジェクトのため）

**適用境界**: 「PR 開発中の Plans.md sync」と「post-merge archive」の 2 段に分けて運用、混同しない。

**関連:**
- 旧本文 (2026-04-22 → 2026-05-13 廃止)
- `feedback-self-trace-expression-breaks-sync-loop.md` — sync loop 回避
- `feedback-active-plan-in-docs.md` — プランは docs/plans/、~/.claude/plans/ 不使用
- 本 memory 上書き先行例: `docs/plans/2026-05-13-phase-2-ui-07.md` Step 0 + commit 6
```
