## 1. Context

### なぜこの PR が必要か

PR A で Prettier が入り「スタイルの機械化」は完了した。しかし Prettier は構文フォーマッタであって **未使用変数・型の緩み・React Hooks 誤用・アクセシビリティ違反** は検出できない。

フロントエンド src-ui/ は現在:

- ESLint **未導入**（PR 品質ゲートが手動レビューのみ）
- `typecheck` が `npm run build` に組み込まれていて **独立実行不可**（CI で分割できず、ローカルでも build 時間を待たされる）
- `tsconfig.json` に `strict: true` 有効だが **type-aware な lint 層が無い**
- 自動生成ファイル（`src/routeTree.gen.ts`, `src/lib/bindings.ts`）は現状 ignore 機構が Prettier のみ
- `package.json` の `format` / `format:check` が `prettier . --write/--check` で glob 非明示（Codex PR #44 任意改善指摘）

PR C で lefthook + CI 強制に繋げる前提として、本 PR で **静的解析の土台** を仕込む。

### この PR の成果（Definition of Done）

1. `npm run lint` が 0 error / 0 warning で通る（既存 25 ファイルに対して）
2. `npm run typecheck` が独立スクリプトとして存在し、`tsc --noEmit` のみを実行
3. `npm run build` から `tsc --noEmit` を外し、vite build 専念
4. `npm run format` / `format:check` の glob が明示化され、対象が文字列で可視化される
5. eslint-plugin-react / react-hooks / jsx-a11y / @typescript-eslint/strict-type-checked が有効で、React 19 + TanStack Router + shadcn/ui コード体系と衝突しない
6. eslint-config-prettier で Prettier とのルール競合が無効化される
7. 自動生成ファイル（routeTree.gen.ts / bindings.ts）は ESLint ignore 対象
8. CI の frontend ジョブは **従来通り build のみ**（lint / typecheck ジョブ追加は PR C）
