# Project Profile: inventory-system

AI Quality Workflow を inventory-system に適用するための project-local profile。
共通 workflow の説明ではなく、この在庫管理アプリの成果物、契約、データ境界、品質ゲートだけを記録する。

## Project Type

- Single-store local desktop inventory management application for a physical handicraft shop.
- Stack: Tauri 2 + React 19 + TypeScript + Rust + SQLite.
- Product method: USDM-driven public requirements from `docs/spec/requirements.md` + `docs/spec/requirements-coverage.md` + linked design docs.
- Architecture boundary: `UI -> CMD -> BIZ -> IO/MNT`.
- Runtime domain: local product master, stock movements, POS CSV import, PLU file export, sales reports, stocktake, backup, operation logs, and diagnostic logs.
- Main user: non-IT store operator. The workflow must favor readable Japanese UI, few daily steps, and recoverable operations.

## Stable Contracts

- `UI -> CMD -> BIZ -> IO/MNT` is one-way. UI must not call IO directly.
- CMD layer is thin: type conversion, BIZ call, and response/error conversion only. Business rules stay in BIZ.
- SQLite runs with foreign keys enabled, WAL mode, and busy timeout.
- DB schema, CHECK constraints, FK relationships, indexes, and date formats are contractual. Date-only fields use `YYYY-MM-DD`; datetime fields use local JST `YYYY-MM-DDTHH:MM:SS`.
- Stock consistency is based on `products.stock_quantity` and non-voided `inventory_movements`. Automatic repair is not allowed without user confirmation.
- Inventory operations that write header + items + stock quantity + movement history must be transactional.
- Operation logs are business audit logs; diagnostic logs are developer-facing file logs. They are separate systems.
- CSV import pipeline is `Parse -> Validate -> Preview -> Commit`.
- CSV import rollback uses logical invalidation / voiding, not physical deletion.
- Z004 parser behavior is contractual: CP932 decode, NEL/CRLF/LF/CR normalization, 5-field parse, sale date extraction, JAN suffix normalization, negative returns support.
- Parse failures are logged as operation logs or preview errors according to design; invalid import attempts do not create misleading completed import records.
- Multiple `jan_code` matches resolve by stable ordering, currently `ORDER BY product_code ASC`.
- `pos_stock_sync` decides whether POS sales import moves stock. Frontend must not infer this from stock unit alone.
- Product codes are either JAN-derived codes or department-prefix custom codes. Custom codes are alphabetic-prefix plus sequence and must not collide with numeric JAN codes.
- Fabric stock can use `cm`; display and thresholds must respect `stock_unit`.
- PLU export is app-to-register only. The app cannot confirm register-side reflection.
- `plu_dirty` and `plu_exported_at` track app-side PLU export state.
- PLU export format is CASIO register tool `.txt` / tab-delimited CP932 according to the current CV17 1.1.1 adapter profile. Real register import/apply confirmation remains a hardware gate.
- Report CSV exports use UTF-8 BOM for Excel compatibility.
- `src/lib/bindings.ts` is generated from Rust via tauri-specta and committed. Rust type changes must regenerate and commit this file.
- The temporary `FallbackCommand` / `typedInvoke` migration path was retired during Phase 2 closeout. Frontend command calls use generated `commands.*` plus `unwrapResult`.
- Router/search state is part of UI behavior for implemented screens. Tabs, filters, and selected entities should be URL-reproducible unless a design doc says otherwise.
- TanStack Query invalidation belongs near mutations/hooks, not scattered through presentational components.
- Phase 2+ Japanese-input validation must be checked on Windows native builds because Tauri 2 on Linux has known IME limitations.
- Tests should carry requirement/spec IDs when the touched area has traceability.

## High-risk Changes

R3 examples for this project:

- SQLite schema, migration, CHECK/FK/index, transaction boundary, or stock consistency behavior.
- `inventory_movements`, `sale_records`, `csv_imports`, stocktake, price history, operation log, or backup semantics.
- Z004 parser behavior, product CSV importer behavior, CP932/BOM handling, NEL handling, JAN normalization, negative return handling, duplicate import behavior, or rollback behavior.
- PLU file format, encoding, dirty/exported state updates, 5000 PLU limit handling, or register workflow messaging.
- BIZ service behavior for product, inventory, CSV import, PLU export, sales, stocktake, integrity, backup, or logs.
- Tauri command arguments/return types, `CmdError` shape, tauri-specta command registration, or generated `src/lib/bindings.ts`.
- Report CSV schema, encoding, filename, or export UX contract.
- UI route/search params, daily workflow screens, CSV import wizard states, unsaved/import-in-progress guards, query invalidation, or user-facing terminology.
- Security/capability changes in `src-tauri/capabilities/`, `tauri.conf.json`, env handling, or filesystem access.
- Test/workflow gates that affect what may be merged.

R4 examples for this project:

- Destructive DB deletion, migration rollback, generated data deletion, or irreversible data lifecycle change.
- Backup restore behavior, automatic backup deletion, or scripts that touch user DB/backups/logs.
- Real POS CSV, sales data, price/cost data, receipt images, or store-specific register data exposure.
- PLU operations that could be mistaken for confirmed register-side reflection or that change real register workflow without human approval.
- Secrets, `.env*`, credentials, key/certificate-looking files, auth files, or Windows app/config area handling.
- Force push, branch deletion, `git reset --hard`, `git clean`, or broad cleanup over ignored/local-only data.

## Data Safety Boundary

- `.env`, `.env.local`, `.env.*.local`, credentials, tokens, keys, certificates, cookies, or `auth.json`.
- Local SQLite DB files, backups, restore targets, app data, operation-log exports, and diagnostic log output.
- Real POS CSV files, real PLU export files, sales reports containing store data, and store-specific price/cost data.
- Receipt images and other source-derived user/store media.
- `docs/research/real-csv/*` except `.gitkeep` and `README.md`.

- `.local/`, quality logs, caches, build outputs, `target/`, `src-tauri/target/`, `node_modules/`, `dist/`, and generated temporary artifacts.
- Windows `C:` drive Codex app/config files or other app-owned state.

## Test Design Focus

Project-specific failure modes to test:

- DB migration idempotency, CHECK/FK enforcement, and old DB compatibility.
- Transaction boundaries for header + items + stock quantity + movement history.
- Rollback/void behavior for duplicate CSV import and stock restoration.
- Z004 parsing across CP932, NEL line separators, empty records, invalid numbers, JAN suffix normalization, and negative return rows.
- Product-code collisions, multiple JAN matches, department prefix sequencing, and discontinued product filtering.
- `pos_stock_sync=0` products avoiding POS-driven stock decrement.
- Stock consistency between `products.stock_quantity` and `inventory_movements`.
- PLU file byte/encoding/field contract and dirty/exported state updates.
- Sales aggregation excluding voided rows and preserving daily/monthly semantics.
- UTF-8 BOM report export compatibility with Excel.
- Tauri `CmdError` mapping and frontend user-facing messages.
- tauri-specta bindings drift after Rust DTO or command changes.
- TanStack Router search param compatibility and active navigation behavior.
- TanStack Query invalidation after CSV import, product update, stocktake, and export mutations.
- Import-in-progress, unsaved changes, and browser-back/useBlocker behavior where applicable.
- UI text matching business meaning, especially "商品コード", "売上明細数", "在庫切れ", and "在庫少".
- Operator-facing status indicators must not be color-only. Tests should assert business labels, roles, icons, or values for states such as "在庫切れ" and "在庫少", not only Tailwind color classes.
- Windows native behavior for Japanese input, file dialogs, and L3 operator demo.
- Windows native L3 for operator-facing UI should include readability and state-distinction checks for the actual non-IT store operator.
- Data safety around real POS CSV, sales/cost data, backups, logs, and receipt images.
- Regression tests for accepted P1/P2 review findings should assert value invariants, not only field shape.

## Workflow Notes

- Plan Packet is expected for R2+ active plans.
- Design Phase happens before Plan Packet for R2+ work when source design docs may be affected.
- Plan Packets are implementation planning evidence, not durable design source of truth. Durable design belongs in architecture, DB design, function design, screen/UI design, decision-log, or ADR docs.
- R3/R4 should use a Test Design Matrix and explicit Boundary / Wire Contract.
- R3 should use review-only sub-agent by default; R4 requires review-only sub-agent and human approval. This profile does not itself add enforcement.
- Requirement/spec IDs should be attached to tests or test comments when the touched area has traceability.
- Behavior changes must update the relevant source document in the same change: architecture, DB design, function design, screen design, or UI tech stack.
- Task-local evidence belongs in active/archive plans. Long PR histories should not make `Plans.md` the product truth.
- `docs/PROJECT_HANDOFF.md` may lag live implementation status. Use it for broad context, not current branch truth.
- CI merge evidence uses L0 pre-push, L1 local-ci full, and L2 hosted final. Docs-only is 0 hosted runs; workflow/script and unknown paths route to full gates. Treat local evidence as valid only when its HEAD SHA matches the PR HEAD.
- Keep CMD thin and BIZ authoritative. UI and CMD should not duplicate business rules that belong in BIZ.
- Prefer Windows native verification for Japanese input and operator-facing Phase 2+ UI flows.
- Risk classification is review-assisted and self-reported in this workflow PR. Machine detection for under-classified changes belongs in a future workflow-hardening plan/PR.
- When a design doc changes a stable contract repeated in this profile, update this profile in the same docs change or record why no profile update is needed.
- New machine enforcement belongs in a separate workflow-hardening plan/PR, not in project-profile setup.
