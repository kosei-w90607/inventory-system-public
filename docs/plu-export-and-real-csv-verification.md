## 確認順序

1. 事前準備: レジ型番、PCツール、SDカード、既存PLUバックアップ、テスト商品を揃える。
2. UI-08 dry run: アプリから Full / Diff の PLUファイルを保存し、ファイル内容を検査する。
3. CASIO PCツール確認: 生成 `.txt` を PCツールに読み込ませ、列、文字、税、部門、件数の受理状況を確認する。
4. レジ投入確認: SDカード / PCツール経由で SR-S4000 に反映し、レジ上の商品名、単価、税、部門、スキャンを確認する。
5. 実売上発生: 登録した商品で販売、複数数量、返品、PLU未登録相当のケースを controlled に発生させる。
6. 当日売上CSV取得: 精算後、PCツール経由で Z004 を取得する。可能なら Z001 / Z002 / Z005 も同時取得する。
7. アプリ取込み確認: 実 Z004 をバックアップ済みテストDBに取り込み、売上、在庫変動、返品マイナス、警告を確認する。
8. 証跡整理: 実CSVや実PLUファイルは commit せず、項目形状、件数、ハッシュ、スクリーンショット、手順差分だけを記録する。

2026-07-03 時点では、UI-08 の実装、PLUファイル保存、画面余白/状態表示の修正、保存後未確認状態の復帰 Alert を確認済み、店頭PC / SR-S4000 側では手動変換ファイルによる成功手順を確認済みで、PR #122 最新のアプリ生成 `.txt` を同じ手順へ通した時の外部機器側の再確認だけが残っていた（上記の通り 2026-07-06 に完了）。

今回 PR の Ready / merge 前に見る点は以下。

1. アプリは保存済み未確認状態のままにする。外部機器側の確認が終わるまで、`この書出しを未反映から外す` は押さない。
2. 可能なら、まず CV17 / SR-S4000 側で代表スキャニングPLUを1件登録し、その設定をSDカード経由でPCツールへ読み出してからエクスポートする。得られた `.txt` のヘッダ、列順、メモリNo.、固定値を「正解形状」として匿名化メモに残し、アプリ生成ファイルと照合する。実JAN・商品名・価格を含む実ファイルはrepoに入れない。
3. PR #122 最新のアプリ生成 `.txt` を CV17 1.1.1 / カシオレジスターツールへ取り込む。手動で変換した reduced file は最終gateの代替にしない。
4. レジスターツールでエラーなく取り込めるかを見る。失敗した場合は、実データを残さず、匿名化したエラー文言、列差分、件数だけを記録する。
5. レジスターツール上で、件数、11列ヘッダ、列順、スキャニングコード、商品名の文字化け/切り詰め、単価、課税方式、部門リンクを代表商品で確認する。
6. レジスターツールから SDカードへ PLU 設定を書き出せるかを見る。件数や結果表示がある場合は匿名化して記録する。
7. SR-S4000 で SDカードの PLU 設定を `設定読み` で適用できるかを見る。レジ側エラーが出た場合は、商品名/JAN/価格を含まない形で内容を記録する。
8. レジ上で代表商品を呼出し/スキャンし、商品名、単価、課税方式、部門が見える範囲で期待通りか確認する。
9. ここまで成功した場合だけアプリに戻り、`この書出しを未反映から外す` を押す。未反映件数とホーム通知が減ることを確認する。
10. どこかで失敗した場合は、アプリ側では未反映解除しない。保存済み未確認状態を残すか、`破棄して再書出し` で pending だけを捨て、設計/formatter/adapter の修正後に再書出しする。

実売上を発生させて Z004 に出るか、在庫引落しまでつながるかは Post-PLU Z004 evaluation の範囲とする。今回 PR では、オーナーが明示的に gate を広げない限り、レジスターツール取込み、SDカード書出し、レジ読込み、代表商品のレジ表示/呼出しまでを必須確認にする。

## 2026-07-06 実機確認・実ファイル形状の確定事項

issue #135 で採取した実 Z001/Z002/Z004/Z005 の匿名化解析結果（詳細はローカル `inventory-field-check/summaries/2026-07-06-z00x-shape-analysis.md`）。CASIO adapter facts として記録し、アプリ core への昇格は各設計 PR で判断する。

### 共通形状（Z001/Z002/Z004/Z005）

- CP932 デコード成功、改行は CRLF のみ。NEL（0x85）の行終端は本サンプルには無し（0x85 は CP932 漢字の内部バイトとしてのみ出現）
- 全フィールドがダブルクォート囲み。Z001/Z002 の金額にはクォート内の千位区切りカンマ（`"##,###"`）が実在する
- 冒頭にメタ6行（2列: 管理No. / ファイル / 帳票 / 番号 / 日付 / 時刻）。**日付は5行目**
- `daily_report_parser`（IO-07）はクォート対応分割・カンマ/¥除去・NEL 正規化を実装済みで、本サンプルの形状を全てカバーする

### Z004 の実構造（全 PLU メモリスロットのダンプ）

- Z004 は売上明細ではなく、メモリNo. 1〜5000 の**全スロットを売上有無を問わず出力**する（データ行 5,000）
- コード欄は **14桁固定幅・右側 `E` パディング**: 13桁JAN + `E`（スキャニングPLU）、8桁独自コード + `EEEEEE`（既存別商材 10件）、全ゼロ = コード未設定。従来「JAN末尾E」と記述していた `E` は識別子ではなくパディングである
- No.1〜216 = 通常PLU枠。名称の 214/216 件が `PLU####` 形式の工場デフォルト名・金額全ゼロで、店舗による登録実体はない。通常PLU は番号打ち込み型の登録機能でバーコードを持たない（SR-S4000 取説「PLU の便利な使い方」）。出荷時配分 216 / スキャニング 4,784 は取説記載と一致し、配分変更の設定は取説に見当たらない
- 現行 `z004_parser`（IO-02）は精算日を1行目から抽出する実装のため、実ファイル（メタ6行 + ヘッダ1行、日付はメタ5行目）では「精算日を抽出できません」で安全に停止する。これは PR #125 L3 で既知の layout A 差分（Plans.md Backlog「Z004 parser の layout A 対応」）の実サンプル再確認であり、改修は Post-PLU Z004 evaluation で 29-io §29.4 の 2 layout 知見を IO-02 へ適用する
- 8桁 + `EEEEEE` は E 除去後8桁となり13桁チェックで invalid_jan エラー行になる。全ゼロコード行（通常PLU 含む）は空スロット扱いで skip される（通常PLU は JAN 照合ベースの取込み対象外として意味的にも整合）

### 店舗運用から導いた公開設計前提

- 現行の精算フローは PLU 売りへの依存を前提にしない。アプリの PLU 出力は既存運用を置換せず、商品別売上を取得するための独立した導入単位として扱う
- スキャニング PLU 領域は、書込み前の衝突検査、固定スロット割当、保存後確認を満たす場合に限りアプリが管理する。既存登録を一律に空き扱いしない
- 実機検証用データは空きスロットだけへ追加し、既存登録を上書きしない。検証後の復元・個別削除は operator が選べる回復手順として記録する


### SDカード / PCツール保存領域

- SD 通常領域 `...\XZ\{年}\{月}` は CV17「SDカードからデータを取込む」実行後に空になり、`...\XZ_BKUP\` と PC 側 `EcrDatas` に同内容が保持されるように見える（目視同一確認済み、機械照合は未実施）
- 日報取込み手順書の主手順（SD 直接 / XZ_BKUP / EcrDatas のいずれか）は、アプリ導入後も CV17 日報運用を続けるかのヒアリング結果と XZ_BKUP 保持仕様の裏取りを待って決める

## 実機確認で使うテストデータ

最小構成は以下。

| 商品 | 目的 |
|---|---|
| JANあり、通常税率10%、通常価格の商品 | 基本PLU登録、Z004販売、在庫減算 |
| JANあり、軽減税率8%または非課税の商品 | 課税方式マッピング確認 |
| JANなし、独自コードの商品 | スキャニングPLU出力前に拒否されることと、別運用が必要なことの確認 |
| 長い日本語名、全角カナ、記号、`_` を含む商品 | 商品名加工とPCツール受理確認 |
| 廃番商品 | PLU書出し除外確認 |
| `plu_dirty=1` の商品 | Diff対象確認 |
| `plu_dirty=0` の商品 | Diff除外確認 |

実店舗の本番PLUを壊さないため、初回は営業時間外に小規模差分で行う。Full投入は、バックアップ復元手順とPCツールの受理結果を確認してから実施する。

## 実CSV取得時の確認方法

1. PLU投入済み商品でテスト販売を行う。
2. 同一商品を複数個販売する。
3. 戻モードで返品を行う。
4. 精算後、PCツール経由で Z004 を取得する。
5. 可能なら同じタイミングで Z001 / Z002 / Z005 も取得する。
6. 実ファイルはローカル保管し、Gitには入れない。
7. バックアップ済みテストDBに取り込み、日次売上、月次売上、在庫照会、商品別在庫変動を確認する。
8. 合成fixtureで通していた仮定との差分を `docs/function-design/55-ui-csv-import.md` と parser 実装へ反映する。

## スマホアプリの扱い

店舗では CASIO ECR+ を毎日使っているが、この文書ではスマホアプリを一次CSV取得経路とは扱わない。まずは PCツール / SDカード経由を正とする。

スマホアプリについて確認する目的は以下。

- 売上・精算・商品別売上の閲覧範囲を把握する。
- PCツールで取得した Z004 の合計や商品別数値と一致するか確認する。
- CSV / Excel / 共有機能があるなら、形式、文字コード、項目、Z004との対応を確認し、将来の取込み元候補にできるか判断する。

ECR+ または関連アプリから取得できるファイルが Z004 と異なる場合は、新しい取込み形式として別要求・別設計に分ける。

## 証跡の残し方

- 残す:
  - 手順、日時、使用したレジ型番、PCツール名/バージョン、SDカード型番。
  - 件数、列名、文字コード、改行、エラー文言、画面表示、差分メモ。
  - 実CSVの匿名化した列形状、またはハッシュ。
- 残さない:
  - 実店舗の生CSV。
  - 実JAN / 実商品名 / 実売上金額を含むファイル。
  - レジのバックアップファイル。
  - Windows native DB本体。

実ファイルを使った確認結果は、必要に応じて source doc に事実だけ昇格する。実ファイルそのものは repo 管理外に置く。

## UI-08 着手時の追加判断

UI-08 実装時に次を先に決める。

1. PCツール投入失敗後の再書出し導線: UI-08 Design Phase（2026-07-01）で、PLUファイル生成と未反映解除を分離する方針に決めた。`prepare_plu_export` では `plu_dirty` を残し、保存後に利用者が `confirm_plu_export_saved` を明示実行した時だけ未反映から外す。
2. UI-08 の画面上に「アプリの責務はPLUファイル保存まで、レジ反映はPCツール/実機側で確認」と明示する。`plu_exported_at` はPCツール受理やレジ反映の証明ではない。
3. Full投入前に既存PLUバックアップを促す文言を出す。バックアップファイルや実PLUファイルはrepoに入れない。
4. ECR+ / スマホアプリの確認結果を UI-08 の案内に含めるか。CSV取得経路として使えないなら、確認・照合用としてだけ扱う。
5. 実Z004取得後、CSV取込み仕様の合成fixture前提と差分が出た場合に同じPRで直すか、別PRに分けるか。
